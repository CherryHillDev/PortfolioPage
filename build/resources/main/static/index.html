1. 개발 환경

  • Spring Boot 2.3.8.RELEASE
  • JDK 11
  • Gradle 6.8
  • MySQL 8.0.20 (RDS)
  • RHEL 8.3 (EC2)
  • Bootstrap 4
  • Mustache
  • Jquery 3

2. 파일 구조

3. 데이터베이스 구조


- 게시글, 회원의 2개 테이블을 생성
- 기본키들은 자동 증가되는 Long 타입
- 댓글은 외부 서비스를 이용했기 때문에, 테이블을 따로 만들지 않음

4. 주요 내용

  • JPA 적용(검색, 페이징까지)
    • 일반적인 쿼리는 자동으로 생성되도록 JPA를 이용
    먼저 Board, User 의 Entity 클래스를 생성, 그리고 생성, 수정시간의 Time을 만들어 공통적으로 상속

    사용할 수 있도록 같은 위치에 Repository 를 두었고
    검색과 페이징, 조회수를 올릴 쿼리를 위한 메소드는 따로 선언
    검색 조건에 맞는 메소드가 실행되도록 규칙을 따라 이름을 지음

    그리고 Service 객체에서 이 Repository 의 메소드를 실행

    페이지 요청이 들어오면 Controller 에서는 해당 페이지 번호로 페이징에 필요한 값을 준비
    목록을 가져오도록 JPA 의 Pageable 을 이용했고(페이지, 한번에 보일 개수, 정렬방법을 넘김),
    JSTL 을 쓰지 않고 Mustache 문법을 사용했기 때문에 이에 맞게 Model 객체에 저장할 값을 정리
  • 구글 로그인
    • 회원 가입 기능을 따로 만들지 않고 구글 로그인을 적용
    구글 클라우드 콘솔에서 OAuth2 사용을 위한 클라이언트를 생성함

    그리고 OAuth2UserService 의 loadUser 메소드를 재정의
    이 메소드 안에서 userRequest 를 가지고 세션에 유저 정보를 저장

    email 로 사용자를 구분해서 데이터베이스의 User 테이블에 저장 또는 수정

    권한 설정 클래스의 configure 메소드에서 설정을 정의
    /api/ 로 시작하는 요청을 제외하고는 권한 없이 접근 가능하지만
    작성, 수정, 삭제는 로그인이 된 상태에서 가능
  • 게시글 CRUD
    • 로그인 후 세션에 user가 저장되어 있으면 글쓰기가 가능

    IndexController 에서 boardController 로 요청을 넘기고, 여기에서 Service 의 메소드 실행
    삭제, 수정도 마찬가지이며 요청 방식에 따라 매핑됨

    삭제, 수정은 세션의 user와 글쓴이가 동일한 경우에만 가능하도록 자바스크립트에서 판단

    게시글 작성, 수정 페이지의 textarea에 ckeditor 적용
  • Utterance 적용
    • 댓글 기능을 따로 만들지 않고 깃허브 댓글 서비스인 Utterance를 적용
    게시글 하나가 깃허브의 이슈 하나가 되어, 댓글을 달면 이슈에 댓글이 달리는 방식
    pathname을 기준으로 이슈가 만들어지게 설정(예 : /board/get/1)
  • 클라우드에 프로젝트 업로드
    • 깃허브에 푸시하면 Travis CI로 빌드 테스트

    성공하면 깃허브 pull, 빌드, 서버 동작을 적은 쉘 스크립트를 직접 실행
    깃허브에서 프로젝트 받음 -> gradlew 를 포함해 실행권한 부여 -> 빌드 파일 생성 -> 실행

5. 보완할 점

  • EC2에서 빌드 시 이따금 멈추는 현상을 해결하지 못함. 프리 티어 메모리 1GB로, 용량이 작아서 일어나는 일으로 추측
  • 구글 로그인만 적용한 점. 자체 회원가입을 만들거나 또는 다른 OAuth2 로그인을 추가하면 좋을 것 같다.
  • ckeditor 에디터의 파일 첨부 기능을 활용하지 못했다. 파일을 클라우드에 저장하고 다운받을 수 있는 기능 추가가 필요하다.

1. 개발 환경

  • Spring 5.2
  • JDK 11

  • Mybatis
  • MySQL 8.0.20 (RDS)
  • RHEL 8.3 (EC2)
  • Bootstrap 4
  • Jquery 3

2. 파일 구조

3. 데이터베이스 구조

- 게시글, 댓글, 회원의 3개 테이블을 생성
- 기본키들은 자동 증가되는 int 타입
- 생성, 수정시간은 데이터의 생성, 수정 시 자동으로 정해지게 함
- 댓글 테이블은 그룹, 그룹 내 순서 컬럼을 만들어 대댓글을 구분해 저장할 수 있게 함

4. 주요 내용

  • 회원가입 유효성 검사
    • 키가 입력될 때마다 검사해서, 유효함을 바로 알 수 있게 함
    아이디는 키가 입력될 때마다 ajax로 중복 여부를 판단해 유효함을 표시
    비밀번호 확인은 비밀번호와 같을 때 유효함을 표시
    이메일은 정규표현식에 일치할 때 인증번호를 보내는 버튼 활성화

    인증번호 전송을 누르면 인증번호를 보내는 중임을 알리는 팝업이 뜨고 아래 인증번호를 입력하는 칸 활성화
    javamail과 AWS SES를 이용해 메일 전송
    (Gmail SMTP 서버를 사용하려 했지만 EC2 환경에서는 SES SMTP 사용이 권장되어서 변경)

    전송한 인증번호가 입력된 경우 유효함이 표시되고
    폼에 변경이 있을 때마다 검사해 모든 입력칸이 유효할 때 계정 생성 버튼 활성화
    계정이 생성되면 로그인 페이지로 이동
  • 전체 게시글, 댓글수, 조회수 표시
    • 게시글 정보와 함께 댓글수, 조회수 표시
    넘겨받은 게시글 목록을 jstl의 반복문으로 나타냄
  • 검색

    • Mybatis의 if 동적 쿼리로 조건에 맞는 게시글을 가져옴
    조건(condition)과 검색단어(keyword)의 초기값은 title, 빈 문자열이며 전체 게시글을 가져옴
    Criteria 객체는 게시글 목록을 가져오는 getBoardList 함수의 매개변수로
    클릭한 페이지, 검색 조건이 담겨있음
  • 페이징
    • 한 페이지 당 10개씩의 게시글을 표시
    컨트롤러에서 조건이 담긴 Criteria를 가지고 페이징에 필요한 변수들을 가진 Paging 객체를 만듦
    MySQL의 LIMIT를 사용해 startPage부터 10개씩 가져옴
    startPage를 포함한 필요한 변수들은 Paging 객체에서 전체 게시글 수인 totalCount 를 바탕으로 계산
    이전, 다음 버튼의 활성화 여부를 위한 prev, next 변수도 계산
    조건에 따라 페이징 버튼 표시
  • 스프링 시큐리티의 요청 인터셉트
    • 로그인 여부에 따라 페이지 접근을 제어하기 위해 사용
    insert, delete, update 로 시작하는 요청을 제외한 나머지는 모두 비인증 상태에서 접근 가능

    커스텀 로그인페이지 경로 설정(login-page), 로그인 후 게시글 화면으로 이동(default-target-url)
    로그인에 실패하면 다시 로그인 페이지를 보이게 함(authentication-failure-url)

    로그아웃 후 게시글 화면으로 이동(logout-success-url)
    또한 허가되지 않은 접근인 경우 에러 페이지로 이동

    데이터베이스에서 유저 정보를 가져오기 위해
    스프링 시큐리티의 UserDetailsService를 구현한 CustomUserDetailsService를 만들고 등록

    CustomUserDetailsService에서는 주어진 username으로 유저 정보를 가져오고
    userDetails 객체에 담아 리턴
  • 게시글 CRUD
    • 글쓰기 폼에는 ckeditor를 적용


    writer가 Authentication에 저장된 username과 같으면 글쓰기를 진행
    수정, 삭제 요청도 같은 과정을 거침

    Authentication에 저장된 유저 정보의 username을 가져와서
    현재 게시글 페이지의 writer와 같으면 수정, 삭제 버튼을 표시
  • 댓글 CRUD
    • 게시글을 가져오면서 게시글 번호에 해당하는 댓글 목록을 가져옴
    대댓글은 부모댓글의 아래에 순서대로 표시되고,
    일반 댓글에만 대댓글을 달 수 있게 버튼을 표시

    게시글과 마찬가지로 writer가 Authentication에 저장된 username과 같은 경우에만
    수정, 삭제를 진행 수정, 대댓글을 클릭하면 바로 아래에 textarea와 버튼을 나타내서 Form 전송이 가능해 짐
  • 클라우드에 프로젝트 업로드
    • 서버가 계속 동작하도록 AWS EC2에 프로젝트를 업로드함 자바, 톰캣 설치 후 WinSCP 프로그램으로 프로젝트 war 파일을 EC2의 톰캣 폴더에 옮김

5. 보완할 점

  • 스프링 시큐리티를 단순하게만 적용했다. 권한 세분화 등 여러 기능을 사용해 볼 수 있을 것 같다.
  • 구글 로그인만 적용한 점. 자체 회원가입을 만들거나 또는 다른 OAuth2 로그인을 추가하면 좋을 것 같다.
  • ckeditor 에디터의 파일 첨부 기능을 활용하지 못했다. 파일을 클라우드에 저장하고 다운받을 수 있는 기능 추가가 필요하다.
